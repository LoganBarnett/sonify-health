** Fixes

* v0.17.0
** Added
1. Windows release binaries.  Releases now attach native Windows executables for
   both ~x86_64~ and ~aarch64~, cross-compiled on Linux via ~llvm-mingw~ (the
   ~*-pc-windows-gnullvm~ triples), so a download runs on a stock Windows
   install with no Cygwin/MSYS2 layer.  The audio stack cross-compiles
   unchanged: ~cpal~ selects its WASAPI backend, a ~windows-sys~ FFI binding
   that needs no SDK.  The x86_64 binary is smoke-tested under ~wine~ in CI; the
   aarch64 binary is build-verified.
** Maintenance
1. Synced the template to v0.39.0 and bumped ~rust-template-foundation~ from
   0.34.0 to 0.39.0.  The bump carries the foundation crate's migration to
   ~sd-notify~ 0.5, and the flake now exposes the template's Windows
   cross-compilation helpers that back the release binaries above.

* v0.14.0
** Maintenance
1. Upgraded ~cpal~ from 0.17 to 0.18.  The 0.18 release folds the per-operation
   audio error types into one ~cpal::Error~/~ErrorKind~, makes
   ~build_output_stream~ take its ~StreamConfig~ by value, and privatizes
   ~DeviceId~'s fields behind ~id()~/~host()~ accessors, so the audio backend
   was updated to match.  Behavior is unchanged — the mixer already called
   ~play()~ explicitly, which 0.18 now requires.
* v0.13.0
** Fixes
1. Releases publish again, and the binary crates leave crates.io.  The v0.12.0
   publish failed because the server crate embeds the dashboard and so cannot be
   packaged for crates.io.  Both ~sonify-health-server~ and ~sonify-health-cli~
   are now unpublished — binary applications shipped as self-contained release
   binaries and via ~nix run~, not crates.io source — so crates.io carries only
   the libraries, ~sonify-health-lib~ and ~sonify-health-voice-derive~.
* v0.12.0
** Added
1. The web dashboard is now embedded in the server binary.  A downloaded release
   runs the full UI with nothing alongside it — no separate asset directory and
   no ~--frontend-path~.  Development builds still read the assets live from
   ~frontend/public~, so ~elm make~ output is picked up without recompiling the
   server.
** Breaking
1. Removed the ~frontend_path~ config field and the ~--frontend-path~ flag.  The
   dashboard is embedded in the binary now (see Added), so there is no asset
   directory to point at; configs and unit files that set either must drop it.
* v0.11.0
** Maintenance
1. Realigned the foundation runtime crate with the build infrastructure and
   synced the template to v0.33.0.  ~rust-template-foundation~ had drifted to
   0.13.0 while the Nix flake tracked a far newer template; both now pin
   v0.33.0, so the shipped binaries run current foundation code.  The sync also
   adds a release-build guard that the cross-compiled arm64 macOS binaries carry
   a valid ad-hoc signature — the kernel SIGKILLs an invalidly-signed one with
   no output.

* v0.4.0
** Added
1. A ~nix run .#quickstart~ app for anyone who has Nix: it runs the daemon
   against the Star Trek bridge preset with zero flags, laying down a writable
   ~config.toml~ in the working directory on first run so edits persist.
** Fixes
1. Linux release downloads now run on non-Nix hosts.  The published
   ~x86_64-linux-gnu~ and ~aarch64-linux-gnu~ binaries are portable dynamic
   glibc builds (~mkGnuPortablePackages~, zig against a glibc 2.17 floor) that
   link the host's ~libasound~, so they play through a desktop's PulseAudio or
   PipeWire; the static ~-musl~ builds remain the headless option.
* v0.3.0
* v0.2.0
** Added
1. A heavy sync with ~rust-template~ brings in a load of shared behavior as well
   as this very change log.
2. Releases now ship prebuilt ~sonify-health~ and ~sonify-health-server~
   binaries for Linux (glibc and static musl) and macOS (Intel and Apple
   Silicon), attached to each GitHub release with a SHA-256 digest and a
   build-provenance attestation.  A one-command ~curl ... | sh~ quick start
   (~install.sh~) downloads the latest release for the platform, lays down the
   Star Trek bridge preset, and runs it — a working, zero-configuration first
   run.  The README also documents the underlying ~curl~ download, ~cargo
   install~, and Nix routes.
** Fixes
1. The daemon now recovers from a silently stalled audio stream.  After a macOS
   output-device change that left the render callback un-pumped without a cpal
   error, playback would stay silent indefinitely; a liveness watchdog now
   detects the stall and rebuilds the stream on the current default device.
   Callback health is exposed via the new ~audio_seconds_since_last_callback~
   and ~audio_callback_stalls_total~ metrics.
