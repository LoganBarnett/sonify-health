* Introduction

sonify-health turns infrastructure health into ambient sound.  Each
machine self-reports via a heartbeat model: probe commands run
periodically, and their results select patches from a sound library.
Each heartbeat can contain multiple notes, and each note maps a metric
to patches via discrete thresholds or smooth gradients.

Heartbeats support three playback modes:

- *Clock* — one-shot plays on a wall-clock cycle, enabling
  time-slotted multi-machine ensembles.
- *Loop* — repeats a phrase with a configurable gap.
- *Continuous* — streams audio that crossfades on metric changes,
  useful for stress layers like CPU or memory pressure.

The result is a Star Trek bridge feel: rhythmic pulses and tonal
shifts that encode real meaning into peripheral awareness without
requiring visual attention.  Physical speaker placement gives spatial
location for free.

A single ~sonify-health~ process can also subscribe to *other*
instances' state and play their heartbeats on its own speakers — so
a speakerless server can run the daemon to do the polling, and a
laptop or workstation elsewhere can listen.  See /Multi-instance
setup/ below.

See [[file:sonification.org][sonification.org]] for the full design rationale.

* Usage

** Preview a patch

Play a named patch from the built-in library:

#+begin_src sh :results none :exports code
sonify-health preview --patch-name sine
sonify-health preview --patch-name alarm --freq 880
sonify-health preview --continuous --patch-name warm
#+end_src

Any patch parameter can be overridden on the command line.  Run
=sonify-health preview --help= for the full list.

** Print the patch library

Export the patch library in a paste-ready format:

#+begin_src sh :results none :exports code
sonify-health print --format toml
sonify-health print --format nix
sonify-health print --format cli --patch-name sine
#+end_src

** Run the daemon

Long-running mode that spawns probe commands, updates shared state,
plays heartbeats, and serves the web frontend:

#+begin_src sh :results none :exports code
sonify-health --config config.toml daemon
#+end_src

Stop with =SIGINT= (Ctrl-C) or =SIGTERM=.

Note that global options like =--config=, =--headless=, and
=--source= must appear *before* the subcommand.

** Multi-instance setup

The daemon decouples the *poller* (the process running probe
commands and producing metric data) from the *renderer* (the
process that owns the audio device).  The two halves don't have
to live on the same machine.

Two independent knobs:

- =headless= is a property of /this/ host — "no speakers, don't
  open an audio device."  It has nothing to do with whether
  anyone subscribes to this instance.
- =[[sources]]= declares /other/ instances /this/ host should
  subscribe to.  Whether subscribed remotes are played locally
  is per-source via =playback_enabled=.

These compose freely: a non-headless instance can both play its
own heartbeats and subscribe to remotes; a headless instance can
declare =[[sources]]= but won't play them (it has no audio
device).  The two-host laptop/server pattern below is just the
common case.

*** Speakerless host (headless)

A host without speakers runs the daemon in headless mode:

#+begin_src sh :results none :exports code
sonify-health --config config.toml --headless daemon
#+end_src

Or set =headless = true= in =config.toml=, or
=services.sonify-health.headless = true= in the NixOS / nix-darwin
module.  No audio device is opened and no play threads are
spawned (for local heartbeats /or/ for any subscribed remotes).
Heartbeat polling and the WebSocket / metrics endpoints stay live
so other instances can subscribe.

A host /with/ speakers stays at =headless = false= even when
other instances subscribe to it — both ends will play, which is
fine if that's what you want.  The listener-side
=playback_enabled= toggle is the right knob for "don't double up
the audio."

*** Subscribing to another instance

A listener declares each remote in its config under =[[sources]]=:

#+begin_src toml :exports code
[[sources]]
name = "prod-db-1"
url  = "wss://db1.internal.example.com/ws"
playback_enabled = true
#+end_src

Or on the command line, repeatable and merged with the config
file's entries:

#+begin_src sh :results none :exports code
sonify-health --source prod-db-1=wss://db1.internal.example.com/ws daemon
#+end_src

Each source carries a =playback_enabled= toggle.  When false, the
listener mirrors state silently — useful when you're back at a
desk where another instance is already sounding the same servers.
Flip it to true on the road and the listener plays the remote
heartbeats from its own speakers without any other change.

The Local Source is always named =localhost= (reserved).  Remote
source names must be unique within the listener's config.

*** Trusted networks only

Outbound connections are not authenticated yet — use the feature
on loopback, a VPN, an SSH tunnel, or a firewalled LAN.  Don't
expose a remote source endpoint on the open internet without
fronting it with your own auth proxy.

** Reverse proxy with WebSocket support

The daemon serves the WebSocket on the same listener as the HTTP
frontend — clients hit the same host:port and either get the
static SPA or trigger a WebSocket upgrade depending on the
request headers.  Reverse-proxying is therefore plain HTTP
proxying /plus/ a tiny dance to forward the =Upgrade= and
=Connection= headers.

By default nginx /strips/ those headers, so a vanilla =proxy_pass=
will silently drop WebSocket traffic — clients see a 200 with the
SPA HTML instead of a 101 protocol-switch.  The fix is the same
whether you're configuring nginx by hand or through NixOS.

*** NixOS

The NixOS sonify-health module defaults to a Unix domain socket
at =/run/sonify-health/sonify-health.sock=, which is the easiest
shape to front with nginx.  Add the nginx user to the
sonify-health group so it can connect to the socket:

#+begin_src nix :results none :exports code
{
  services.sonify-health = {
    enable = true;
    # Default: socket = "/run/sonify-health/sonify-health.sock";
  };
  users.users.nginx.extraGroups = [config.services.sonify-health.group];

  services.nginx = {
    enable = true;
    recommendedProxySettings = true;
    recommendedTlsSettings = true;

    virtualHosts."sonify.example.com" = {
      enableACME = true;
      forceSSL = true;
      locations."/" = {
        proxyPass = "http://unix:${config.services.sonify-health.socket}:";
        proxyWebsockets = true;
      };
    };
  };
}
#+end_src

=proxyWebsockets = true= is the nixpkgs convenience that wires
=Upgrade= / =Connection= headers /and/ the required
=$connection_upgrade= map.  Listeners point at
=wss://sonify.example.com/ws=.

For TCP mode instead of a socket, set
=services.sonify-health.socket = null;= and =port = 3000;=, and
change the =proxyPass= to =http://127.0.0.1:3000=.

*** Plain nginx

If you're not on NixOS, the same pattern in flat nginx config:

#+begin_src nginx :exports code
# In the http block — the $connection_upgrade variable is not
# predefined.
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

server {
    listen 443 ssl http2;
    server_name sonify.example.com;

    ssl_certificate     /etc/letsencrypt/live/sonify.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/sonify.example.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade    $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host       $host;
    }
}
#+end_src

Listeners use =wss://sonify.example.com/ws= the same way.  The
daemon doesn't know or care that TLS is being terminated upstream
— it sees a regular HTTP/1.1 connection with =Upgrade:
websocket= and runs the handshake.

The default 60s =proxy_read_timeout= is left in place
deliberately: the daemon broadcasts =metric_changed= every poll
interval, so a healthy WebSocket sees traffic well inside that
window.  If a connection /does/ idle out, something has actually
gone wrong (the remote's polls have stopped, or the network has
silently dropped) — a reconnect is the right response.

* Configuration

Copy =config.example.toml= to =config.toml=.  All fields are
optional; the daemon runs with sensible defaults.

Top-level keys:

- =log_level= / =log_format= — logging control.
- =listen= — web server bind address.
- =audio_device= — audio output device name (default: system default).
- =headless= — when true, skip the audio device entirely.  Pollers
  and the WebSocket stay live; no audio is produced.  Default
  false.  See /Multi-instance setup/ above.
- =[patches]= — named patch definitions for the sound library.
- =[[sources]]= — remote ~sonify-health~ instances to subscribe
  to.  Each entry sets =name=, =url=, and an optional
  =playback_enabled= (default false).

Each =[[heartbeats]]= entry defines a probe and one or more notes.
The =playback= field selects the mode (=clock=, =loop=, or
=continuous=; defaults to =clock=).  The =result_mode= field
determines how the command output is read:

- =exit-code= — exit code 0/1/2 maps to discrete thresholds.
- =stdout= — command prints a metric value (0.0–1.0) to stdout.

Each note within a heartbeat carries a =transition= that maps the
probe metric to patches:

- =discrete= — selects the first state whose =threshold= exceeds the
  metric.
- =gradient= — interpolates between adjacent patches.

* Gotchas

- *Audio device access* — the daemon opens the default audio output.
  On Linux this requires ALSA or PipeWire.  On macOS, CoreAudio is
  used directly.
- *NTP precision* — multi-machine timing relies on wall-clock
  synchronization.  NTP provides ~10 ms accuracy, sufficient for
  the default cycle duration.
- *Nix sandbox* — =nix build= runs tests in a sandbox without audio
  hardware.  Integration tests that play sound are skipped in the
  sandbox build.
- *Global options* — flags like =-c=, =--log-level=, and =--listen=
  must appear before the subcommand, not after it.
