#+title:     Contributing to sonify-health
#+language:  en

Guidelines for contributing to sonify-health.

* Document conventions

- *Root-level UPPER_CASE names* are reserved for the standard set: =README=,
  =CONTRIBUTING=, =LICENSE=.  All other documentation lives in =docs/= with
  lower-kebab-case file names (e.g. =docs/systemd.org=).
- Every document must be reachable — directly or indirectly — from =README.org=.
- Use org-mode for all new documents.

* Development environment

Enter the Nix dev shell before running any commands:

#+begin_src sh :results none :exports code
nix develop
#+end_src

Or, if you use direnv:

#+begin_src sh :results none :exports code
direnv allow
#+end_src

* Code style

** Semantic error handling

This project uses *semantic error types* with ~thiserror~ rather than generic
error wrapping with ~anyhow~.  Every error variant must include sufficient
context to diagnose the problem without access to the source code or the running
environment.

See ~crates/cli/src/config.rs~ for examples of well-structured error types.

** Comments

Comments must be complete sentences, wrapped at 80 columns, with standard
punctuation.  Only comment non-obvious intent, invariants, tradeoffs, or
historical constraints; do not restate control flow.

** Initialisms

Use Pascal initialisms (~Url~) instead of preserved initialisms (~URL~) when
authoring new identifiers in camel case.

* Error handling

This project uses *semantic error types* with ~thiserror~ rather than generic
error wrapping with ~anyhow~.

** Why not ~anyhow~?

While ~anyhow~ is convenient, it encourages lazy error handling that loses
important context.  When an error occurs in production you often only have logs
to work with.  A bare "No such file or directory" message with no indication of
/which/ file or /what/ operation failed is not actionable.

** Core principle: always add context

All errors must include sufficient context to diagnose the problem without
access to the source code or the running environment.

Use semantic error types that explain what went wrong and why:

#+begin_src rust :results none :exports code
use thiserror::Error;

#[derive(Debug, Error)]
pub enum ConfigError {
    #[error("Failed to read configuration file at {path:?} \
             during startup: {source}")]
    FileRead {
        path: PathBuf,
        #[source]
        source: std::io::Error,
    },

    #[error("Failed to parse configuration file at {path:?}: \
             {source}")]
    Parse {
        path: PathBuf,
        #[source]
        source: toml::de::Error,
    },

    #[error("Configuration validation failed: {0}")]
    Validation(String),
}
#+end_src

** Guidelines

- Every error variant should explain what operation failed and include relevant
  data (file paths, addresses, etc.).
- Use ~#[source]~ to preserve the full error chain.
- For file operations, include the path and (where useful) the current working
  directory.
- Never log sensitive data (passwords, tokens, connection strings with
  credentials).
- ~anyhow~ is acceptable only in throwaway prototypes.  Once you know what
  errors can occur, convert to semantic types.

** Error type hierarchy

Structure error types in layers:

1. *Low-level* (libraries/modules) — specific to that component.
2. *Mid-level* (business logic) — domain-specific.
3. *Top-level* (main.rs) — application lifecycle.

Each layer adds context about what the application was trying to do.

* CSS

** Flat selectors only

Each selector targets exactly one element type.  No nesting, no combinators, no
~>~ or ~+~ chains.  If an element needs unique styling, use a class or ID — not
a compound selector.

Good:

#+begin_src css :results none :exports code
nav {
  margin-bottom: 1rem;
}
#+end_src

Bad — nested combinator creates implicit coupling between elements:

#+begin_src css :results none :exports code
.container > nav {
  margin-bottom: 1rem;
}
#+end_src

** CSS variables for shared values

Use CSS custom properties (variables) defined on ~:root~ for any value shared
between selectors.  Never hard-code colours, spacing scales, or font stacks
directly in a selector.

Good:

#+begin_src css :results none :exports code
:root {
  --color-text: #111111;
}

p {
  color: var(--color-text);
}

h1 {
  color: var(--color-text);
}
#+end_src

Bad — duplicated literal makes updates error-prone:

#+begin_src css :results none :exports code
p {
  color: #111111;
}

h1 {
  color: #111111;
}
#+end_src

** Light and dark mode

Provide light-mode values in ~:root~ and override them inside ~@media
(prefers-color-scheme: dark)~.  Never create separate selectors per colour
scheme — the variables handle the switch.

Good:

#+begin_src css :results none :exports code
:root {
  --color-bg: #ffffff;
}

@media (prefers-color-scheme: dark) {
  :root {
    --color-bg: #111111;
  }
}

body {
  background: var(--color-bg);
}
#+end_src

Bad — duplicated selector with hard-coded colours:

#+begin_src css :results none :exports code
body {
  background: #ffffff;
}

@media (prefers-color-scheme: dark) {
  body {
    background: #111111;
  }
}
#+end_src

** No inline styles in Elm

All presentation belongs in the stylesheet.  Elm views must not use ~style~
attributes.  Use semantic HTML elements (~main_~, ~nav~, ~header~, ~section~) so
that element selectors in CSS are sufficient and classes are rarely needed.

Good:

#+begin_src elm :results none :exports code
view model =
    main_ []
        [ nav [] [ ... ]
        , viewPage model
        ]
#+end_src

Bad — inline styles bypass the stylesheet and cannot respond to dark mode, media
queries, or future design changes:

#+begin_src elm :results none :exports code
view model =
    div [ style "padding" "2rem" ]
        [ nav [ style "margin-bottom" "1rem" ] [ ... ]
        , viewPage model
        ]
#+end_src

* Systemd secrets

The NixOS module uses systemd's ~LoadCredential~ to pass secret files to the
service without granting the service user direct read access.  See
=docs/systemd.org= for the full pattern.

* Logging levels

Use appropriate log levels for different severity:

- ~error!()~ — errors that prevent normal operation.
- ~warn!()~ — recoverable issues or degraded functionality.
- ~info!()~ — important state changes.
- ~debug!()~ — detailed diagnostic information.
- ~trace!()~ — very verbose debugging.

Never log sensitive data (passwords, tokens, secrets).

* Formatting

Run ~cargo fmt~ before committing.  The pre-commit hook enforces this
automatically, so commits made outside the Nix shell may be rejected if
~rustfmt~ is not on ~PATH~.

* Testing

Run the full test suite after every change:

#+begin_src sh :results none :exports code
nix develop --command cargo test --workspace
#+end_src

Tests must exit with a proper error code.

* UI philosophy: no modals

Do not use modals anywhere in the UI.  Replace every modal with an inline
expansion that keeps surrounding context visible.

** Why

Modals block the UI — which may contain information relevant to the decision at
hand — and they are an accessibility nightmare (backdrop divs, tab-index
trapping, z-index layering).  As an interface matures, the probability of
needing a modal inside a modal trends to 1.  Powerful UIs do not use modals.

** Patterns

- *Export / large output*: clicking the trigger button slides the surrounding
  content down and reveals the output inline, in the normal document flow.
- *Destructive confirmations*: replace the triggering button in-place with a
  "Confirm delete?  Yes / No" prompt.  When the user decides (or clicks
  elsewhere), restore the original button.
- *Progressive disclosure*: expand a section in-place rather than overlaying.

Every piece of revealed UI must live in the normal document flow.  No backdrop
divs, no tab-index trapping, no z-index layering.

* UI philosophy: every label earns a help entry

The dashboard carries a lot of specialised vocabulary — metric tiers, patches,
notes, playback modes, transitions — and a contributor (or a returning user)
cannot realistically remember what each one means.  The bottom-of-page Help
panel exists to answer that question in place, without navigating away.

** Definition of Done for UI changes

Any new or renamed UI label, section header, or control caption is not
considered done until it has a corresponding entry in ~frontend/src/Help.elm~.
That means:

1. Pick a stable kebab-case ~Key~ (e.g. ~"metric-tiers"~, ~"patch-detune"~).
   Prefer names that survive cosmetic renames.
2. Add a case to ~Help.lookup~ returning one or two sentences.  Keep it short —
   the panel is small by design.  Longer explanations belong in ~docs/~.
3. Render the label with ~class "help-label"~ on a ~button~ element (not a plain
   ~span~ or ~label~) so keyboard and screen-reader users get proper semantics
   for free.  Wire its ~onClick~ to ~HelpClicked "your-key"~.

If a label has no explanation beyond "it does what it says," that is still a
one-sentence help entry — write it anyway, because what seems obvious to the
author is rarely obvious to a newcomer.

* Commit workflow

1. Make your changes on a feature branch.
2. Ensure ~cargo fmt~ and ~cargo clippy --workspace~ produce no warnings.
3. Run ~cargo test --workspace~ and confirm all tests pass.
4. Open a pull request against ~main~.

* Release binaries

A release builds your binary crates for the mainstream targets and attaches them
to the GitHub release.  The macOS binaries are *cross-compiled on Linux* (via
zig) rather than on a macOS runner, so a release never waits on scarce macOS CI
capacity.  This is automatic — ~mkDarwinCrossPackages~ is already wired into
~flake.nix~ and needs no configuration.

The one exception is a crate that links Apple frameworks — anything that
(transitively) pulls ~cpal~, ~objc2-*~, ~core-foundation~, ~security-framework~,
~coreaudio-sys~, and similar.  This is not CLI-specific; the ~server~ crate can
link them just as readily.  Such a crate needs Apple's framework headers and
link stubs, which come from the (unfree) Apple SDK.  To enable it, touch two
spots in ~flake.nix~:

1. add ~config.allowUnfree = true~ and ~config.allowUnsupportedSystem = true~ to
   the ~nixpkgs~ import (the latter is needed whenever that ~pkgs~ is
   non-darwin, i.e. in Linux CI), because ~apple-sdk~ is unfree and
   darwin-gated; and
2. pass ~appleSdk = pkgs.apple-sdk.src~ to the ~mkDarwinCrossPackages~ call.

~apple-sdk.src~ is a plain fetch of the SDK's inert data (headers and ~.tbd~
stubs); the full ~apple-sdk~ package does not build on Linux, but its ~.src~
does.  Keeping the ~allowUnfree~ acceptance in your own ~flake.nix~ is
deliberate — the licence consent is visible in your project, not hidden in the
foundation library.  A pure-libSystem executable does nothing: zig bundles the
libSystem stubs, so it cross-compiles licence-free.

* Changelog conventions

~CHANGELOG.org~'s ~* Upcoming~ section uses three subheadings:

- ~** Added~ — features or behavior added.  This is the loose "what's new"
  section: anything an adopter would describe as "the next release does X now"
  belongs here, even when previous releases also did something different in the
  same surface.
- ~** Fixes~ — bug fixes.
- ~** Breaking~ — backward-incompatible changes.  Required when the ABI check
  proves a public-signature break; recommended whenever behavioral breakage
  shipped that downstream consumers should know about, even when signatures are
  preserved.

A single change may appear under more than one subheading when it fits more than
one.  A fix that also adds an option to opt out of the broken behavior belongs
under both ~** Added~ and ~** Fixes~.

Each entry should be a complete sentence-or-two written for an adopter reading
the release notes, not a reviewer reading the diff.  Name what changed in terms
of the surface the adopter sees, not the implementation.

Write each entry as a numbered list item (~1.~, ~2.~, …) under its subheading,
not a dash bullet.

Every pull request must add at least one entry under ~* Upcoming~; CI rejects a
pull request that adds none.  This keeps the changelog written incrementally,
while the context is fresh, rather than reconstructed from the diff at release
time.
