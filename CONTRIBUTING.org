#+title:     Contributing to sonify-health
#+language:  en

Guidelines for contributing to sonify-health.

* Document conventions

- *Root-level UPPER_CASE names* are reserved for the standard set: =README=,
  =CONTRIBUTING=, =LICENSE=.  All other documentation lives in =docs/= with
  lower-kebab-case file names (e.g. =docs/systemd.org=).
- Every document must be reachable — directly or indirectly — from =README.org=.
- Use org-mode for all new documents.

* Development environment

Enter the Nix dev shell before running any commands:

#+begin_src sh :results none :exports code
nix develop
#+end_src

Or, if you use direnv:

#+begin_src sh :results none :exports code
direnv allow
#+end_src

* Code style

** Semantic error handling

This project uses *semantic error types* with ~thiserror~ rather than
generic error wrapping with ~anyhow~.  Every error variant must include
sufficient context to diagnose the problem without access to the source
code or the running environment.

See ~crates/cli/src/config.rs~ for examples of well-structured error
types.

** Comments

Comments must be complete sentences, wrapped at 80 columns, with standard
punctuation.  Only comment non-obvious intent, invariants, tradeoffs, or
historical constraints; do not restate control flow.

** Initialisms

Use Pascal initialisms (~Url~) instead of preserved initialisms (~URL~) when
authoring new identifiers in camel case.

* Systemd secrets

The NixOS module uses systemd's ~LoadCredential~ to pass secret files to
the service without granting the service user direct read access.  See
=docs/systemd.org= for the full pattern.

* Logging levels

Use appropriate log levels for different severity:

- ~error!()~ — errors that prevent normal operation.
- ~warn!()~  — recoverable issues or degraded functionality.
- ~info!()~  — important state changes.
- ~debug!()~ — detailed diagnostic information.
- ~trace!()~ — very verbose debugging.

Never log sensitive data (passwords, tokens, secrets).

* Formatting

Run ~cargo fmt~ before committing.  The pre-commit hook enforces this
automatically, so commits made outside the Nix shell may be rejected if
~rustfmt~ is not on ~PATH~.

* Testing

Run the full test suite after every change:

#+begin_src sh :results none :exports code
nix develop --command cargo test --workspace
#+end_src

Tests must exit with a proper error code.

* UI philosophy: no modals

Do not use modals anywhere in the UI.  Replace every modal with an
inline expansion that keeps surrounding context visible.

** Why

Modals block the UI — which may contain information relevant to the
decision at hand — and they are an accessibility nightmare (backdrop
divs, tab-index trapping, z-index layering).  As an interface matures,
the probability of needing a modal inside a modal trends to 1.  Powerful
UIs do not use modals.

** Patterns

- *Export / large output*: clicking the trigger button slides the
  surrounding content down and reveals the output inline, in the normal
  document flow.
- *Destructive confirmations*: replace the triggering button in-place
  with a "Confirm delete?  Yes / No" prompt.  When the user decides (or
  clicks elsewhere), restore the original button.
- *Progressive disclosure*: expand a section in-place rather than
  overlaying.

Every piece of revealed UI must live in the normal document flow.  No
backdrop divs, no tab-index trapping, no z-index layering.

* UI philosophy: every label earns a help entry

The dashboard carries a lot of specialised vocabulary — metric
tiers, patches, notes, playback modes, transitions — and a
contributor (or a returning user) cannot realistically remember
what each one means.  The bottom-of-page Help panel exists to
answer that question in place, without navigating away.

** Definition of Done for UI changes

Any new or renamed UI label, section header, or control caption
is not considered done until it has a corresponding entry in
~frontend/src/Help.elm~.  That means:

1. Pick a stable kebab-case ~Key~ (e.g. ~"metric-tiers"~,
   ~"patch-detune"~).  Prefer names that survive cosmetic
   renames.
2. Add a case to ~Help.lookup~ returning one or two sentences.
   Keep it short — the panel is small by design.  Longer
   explanations belong in ~docs/~.
3. Render the label with ~class "help-label"~ on a ~button~
   element (not a plain ~span~ or ~label~) so keyboard and
   screen-reader users get proper semantics for free.  Wire its
   ~onClick~ to ~HelpClicked "your-key"~.

If a label has no explanation beyond "it does what it says," that
is still a one-sentence help entry — write it anyway, because
what seems obvious to the author is rarely obvious to a newcomer.

* Commit workflow

1. Make your changes on a feature branch.
2. Ensure ~cargo fmt~ and ~cargo clippy --workspace~ produce no warnings.
3. Run ~cargo test --workspace~ and confirm all tests pass.
4. Open a pull request against ~main~.
