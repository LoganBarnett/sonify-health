* DONE Configurable labels and colors for metric tiers
The metric status labels (=healthy=, =degraded=, =down=) and their
thresholds are hardcoded in both the backend (=metric_label= in
=preview_state.rs=) and the frontend (=metricClass= / =metricLabel= in
=Main.elm=).  The CSS correctly uses =text-transform: uppercase= for
a11y, so the display is fine — but the label strings and tier boundaries
are not user-configurable and the system supports arbitrary severity
ranges.

Labels are *independent of transitions*.  A heartbeat (or note) can
define labeled tiers at arbitrary severity thresholds regardless of
whether the transition is discrete or gradient.  For gradient
transitions this means labels need not align with the keyframe
patches — you can label any point along the 0–1 range.

Each tier needs:
- A label (arbitrary string, displayed via CSS =text-transform:
  uppercase=).  Default: ={{value}}= — just displays the raw metric
  value, so users who do not care about labels get a sensible
  fallback without extra configuration.
- A color (used for the badge background / accent).
- A threshold defining the upper bound of the tier's range.

The UI must allow editing labels, picking colors, and setting
thresholds for each tier.  Labels and colors must round-trip through
the config file.  If no custom tiers are configured, fall back to
displaying ={{value}}=.

* TODO Eliminate all modals — inline-expand UI pattern
Do not use modals anywhere in the UI.  Replace every modal with an
inline expansion that keeps surrounding context visible.

Design principles:
- The UI never obscures content the user might need while making a
  decision.
- No backdrop divs, no tab-index trapping, no z-index layering.
- Every piece of revealed UI lives in the normal document flow.

Specific patterns to apply:
- *Export*: clicking the export button slides the action bar down
  and reveals the export text below it, inline.
- *Destructive confirmations*: replace the triggering button in-place
  with a "Confirm delete?  Yes / No" label and two buttons.  When
  the user decides (or clicks elsewhere), restore the original
  button.
- *Any future progressive disclosure*: expand a section in-place
  rather than overlaying.

Backport the existing export modal to this inline pattern.

Document this philosophy and the concrete examples in the
contributing guide (=CONTRIBUTING.org= or equivalent).  Also
backport the guideline to the Rust template repository so all
future projects inherit it.

The core argument: modals mark the boundary where the UI ran out of
answers.  As the interface matures, the probability of needing a
modal inside a modal trends to 1.  They block the UI — which may
contain information relevant to the decision at hand — and they are
an accessibility nightmare.  Powerful UIs do not use modals.
